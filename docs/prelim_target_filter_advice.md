# 初賽目標辨識防誤框建議

日期：2026-07-05

## 規則重點

- 初賽飛行展示時間為 10 分鐘，需即時回傳畫面。
- 回傳畫面中要框出辨識成功目標的類別與座標。
- 飛行展示結束後 10 分鐘內，需繳交目標辨識與定位結果的畫面截圖與座標。
- 繳回座標應以 TWD97 呈現。
- 初賽只要在展示時間內正確辨識並定位一個目標，且成功返航，就可進入名次排序。
- 初賽排名採時間優先、精度次之。
- 目標為 3 輛靜止狀態之公路局核可車輛，且大會會在目標上放置指認圖以避免混淆。

## 核心策略

初賽不要把問題當成「找出所有車輛」，而要改成「在既有車輛候選中，只放行白色車輛加指認圖或 X 標記」。

目前主流程可以保持不變：

1. YOLO 或其他 detector 提出車輛候選框。
2. 地圖配準取得影像到參考地圖的轉換。
3. 將車輛中心投影到 WGS84 與 TWD97。
4. 輸出截圖、JSON、CSV 與流程總覽圖。

建議只在 YOLO 後、座標投影前新增一層 `competition_target_filter`，用來判斷候選框是否能被正式回報。

## 建議的過濾條件

候選車框應同時滿足：

- 車體顏色符合白色車輛特徵。
- 車頂或車身區域存在指認圖或 X 標記。
- bounding box 尺寸與長寬比合理，避免大片場景框或多車合併框。
- YOLO 類別優先限制為 `car`，初賽模式下先避免 `bus`、`truck` 造成大框誤判。
- 地圖投影點需落在有效參考地圖範圍內。

## 排序與輸出

不要只用 YOLO confidence 排序。初賽模式建議改用：

```text
final_score = X 標記分數 > 白色車比例 > YOLO 信心分數 > 框尺寸合理性
```

比賽時建議只輸出通過驗證的 top-1 目標。若第二、第三個目標也通過 X 標記與白車驗證，再輸出 top-3。

## 現有流程的風險

- 目前通用車輛偵測會把一般車、非白車或場景中的車都列入候選。
- 低 confidence 門檻有利於提高召回率，但比賽時容易增加誤框。
- 大框可能把多台車或背景物件一起框入，這類框不適合直接交給裁判指認。

## 實作優先順序

1. 建立 X 標記或指認圖 verifier。
2. 建立白色車體比例檢查。
3. 初賽模式預設 `--vehicle-classes car --max-detections 3`。
4. 只把 verified target 畫成正式框；一般 YOLO 候選只做 debug 或不顯示。
5. 保留現有地圖配準與 TWD97 輸出流程。

最重要的判斷：初賽最大風險不是座標格式，而是指認錯目標。寧可少報一個高可信目標，也不要回報多個未驗證車框。

## Real-Time 與 Near-Real-Time 評估

測試日期：2026-07-05

使用現有 `scripts/localize_vehicles.py` 工作流測試單張 `test_image/frame_000161.jpg`：

| 設定 | `detect_vehicles` | total |
| --- | ---: | ---: |
| `yolo26x.pt`，MPS，品質取向設定 | 34.17s | 35.74s |
| `yolo26l.pt`，MPS，`opencv_test.py` 預設模型 | 28.41s | 30.04s |
| `yolo26l.pt`，full-frame 粗篩參數 | 18.50s | 20.09s |

結論：如果每次都呼叫完整 `localize_vehicles.py` 子程序，現有工作流不能 real time，也不算 near real time。瓶頸主要在每張 frame 都重新啟動流程並執行 YOLO 偵測；地圖匹配本身約 1 秒。

同一個 Python 程序內預載 `yolo26l.pt` 後，full-frame `imgsz=640` 的連續推論測試結果：

| 項目 | 時間 |
| --- | ---: |
| model load | 0.10s |
| first predict | 0.36s |
| second predict | 0.17s |

這表示 near-real-time 的關鍵不是模型本身，而是要避免每張 frame 都重啟完整定位腳本。建議架構是讓串流程式常駐載入一個輕量粗篩器，每隔 N frame 或 N 秒快速掃一次；只有粗篩判斷可能有目標車時，才觸發完整定位流程輸出 TWD97、截圖與報告。
